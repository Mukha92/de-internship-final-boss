# Описание скриптов ETL-пайплайна CYBERPIKCHA_2077

В этом документе представлено подробное описание каждого скрипта из папки `scripts/`. Скрипты предназначены для управления всеми этапами ETL-пайплайна: от инициализации базы данных до очистки всех компонентов системы.

---

## 1. 🏛 init_clickhouse.py

**Назначение:** Создание структуры базы данных ClickHouse для хранения сырых данных (raw-слой) и витрин (mart-слой).

### Что делает скрипт

Скрипт последовательно создаёт все необходимые компоненты хранилища данных:

1. **Создание базы данных `raw`** — пространство имён для сырых данных, поступающих из Kafka
2. **Создание таблиц raw-слоя:**
   - `raw.stores` — данные о магазинах (название, сеть, адрес, координаты, режим работы)
   - `raw.products` — информация о товарах (название, категория, КБЖУ, цена, производитель)
   - `raw.customers` — сведения о клиентах (ФИО, контакты, дата рождения, предпочтения)
   - `raw.purchases` — данные о покупках (состав чека, сумма, способ оплаты, доставка)

3. **Создание базы данных `mart`** — пространство имён для аналитических витрин
4. **Создание таблиц mart-слоя:**
   - Измерения (`dim_*`): справочники клиентов, магазинов, продуктов, производителей, дат, адресов, локаций
   - Факты (`fact_*`): факты покупок и позиции чеков
   - Витрина признаков (`customer_features_mart`) — 30 бинарных признаков для ML-кластеризации клиентов

5. **Создание Materialized Views** — материализованные представления для автоматической загрузки данных из raw-слоя в mart-слой при поступлении новых данных

### Режимы работы

- **Полная инициализация** — создание всех баз данных и таблиц (raw + mart)
- **Только raw-слой** — создание только таблиц для сырых данных
- **Только mart-слой** — создание только витрин (требует существующего raw-слоя)
- **Сухой запуск** — показать SQL-команды без фактического выполнения
- **С удалением существующих** — удалить старые таблицы перед созданием новых

### Как работает

1. Подключается к ClickHouse по указанным параметрам (хост, порт, пользователь, пароль)
2. Читает SQL-скрипты из папки `sql/`
3. Выполняет SQL-команды последовательно: создание БД → создание таблиц → создание представлений
4. Выводит сводку о созданных таблицах с количеством строк

### Когда использовать

- При первом развёртывании проекта
- После полной очистки данных (`cleanup_all.py`)
- При необходимости пересоздать структуру хранилища

## 2. 📊 generate_data.py

**Назначение:** Генерация синтетических данных для розничной сети «CYBERPIKCHA 2077».

### Что делает скрипт

Скрипт создаёт реалистичные тестовые данные четырёх сущностей:

1. **Магазины** — генерирует информацию о торговых точках:
   - Уникальный идентификатор и название
   - Торговая сеть («Большая Пикча» или «Малая Пикча»)
   - Тип магазина (гипермаркет, супермаркет, магазин у дома, экспресс)
   - Полный адрес с координатами (широта, долгота)
   - Режим работы, возможность онлайн-заказов и доставки
   - Данные о менеджере магазина

2. **Товары** — создаёт каталог продукции по категориям:
   - Молочные продукты, фрукты/ягоды, овощи/зелень
   - Хлеб/выпечка, мясо/рыба/яйца, напитки, снеки
   - Для каждого товара: название, категория, КБЖУ, цена, данные производителя

3. **Клиенты** — формирует базу покупателей:
   - Персональные данные (ФИО, email, телефон, дата рождения, пол)
   - Адреса (местожительство и доставки)
   - Участие в программе лояльности
   - Предпочтения по магазинам и способам оплаты

4. **Покупки** — генерирует историю транзакций:
   - Уникальный номер покупки и дата/время
   - Ссылки на клиента, магазин, способ оплаты
   - Список товаров в чеке с количеством и ценой
   - Флаг доставки и итоговая сумма

### Как работает

1. Использует библиотеку `Faker` для генерации реалистичных персональных данных
2. Создаёт заранее определённое количество объектов (магазины, товары, клиенты, покупки)
3. Сохраняет каждый объект в отдельный JSON-файл в соответствующую папку:
   - `data/stores/` — файлы магазинов
   - `data/products/` — файлы товаров
   - `data/customers/` — файлы клиентов
   - `data/purchases/` — файлы покупок

### Настройки

Можно указать количество генерируемых объектов:
- Количество магазинов (по умолчанию: 45)
- Количество товаров (по умолчанию: 50)
- Количество клиентов (по умолчанию: равно количеству магазинов)
- Количество покупок (по умолчанию: 200)

### Когда использовать

- При первоначальной настройке пайплайна
- Для создания тестовых данных перед запуском ETL
- При необходимости увеличить объём данных для тестирования производительности

## 3. 🍃 load_to_mongo.py

**Назначение:** Загрузка сгенерированных JSON-данных в базу данных MongoDB.

### Что делает скрипт

Скрипт читает JSON-файлы из директории `data/` и загружает их в коллекции MongoDB:

1. **Подключение к MongoDB** — устанавливает соединение с сервером MongoDB по URI из конфигурации
2. **Очистка коллекций** (опционально) — удаляет существующие документы перед загрузкой новых данных
3. **Загрузка данных по коллекциям:**
   - `stores` ← файлы из `data/stores/`
   - `products` ← файлы из `data/products/`
   - `customers` ← файлы из `data/customers/`
   - `purchases` ← файлы из `data/purchases/`

### Как работает

1. Сканирует указанную директорию на наличие подпапок с данными
2. Для каждой подпапки читает все JSON-файлы
3. Извлекает данные из JSON и вставляет их в соответствующую коллекцию MongoDB
4. Выводит статистику загруженных документов по каждой коллекции

### Режимы работы

- **С очисткой** (по умолчанию) — перед загрузкой удаляет все документы из коллекций
- **Без очистки** (`--no-clear`) — добавляет данные к существующим документам

### Когда использовать

- После генерации данных (`generate_data.py`)
- Перед запуском Producer для передачи данных в Kafka
- При необходимости перезагрузить данные в MongoDB

## 4. 📡 run_producer.py

**Назначение:** Передача данных из MongoDB в Kafka (Producer).

### Что делает скрипт

Скрипт читает данные из коллекций MongoDB и отправляет их в топики Kafka:

1. **Подключение к MongoDB** — соединяется с базой данных для чтения документов
2. **Подключение к Kafka** — устанавливает соединение с брокером Kafka
3. **Чтение данных из MongoDB** — последовательно извлекает документы из каждой коллекции
4. **Анонимизация чувствительных данных** (опционально) — хеширует персональные данные клиентов с помощью HMAC-SHA256:
   - Email адреса
   - Номера телефонов
   - ФИО
5. **Отправка в Kafka** — публикует сообщения в соответствующие топики:
   - `stores` ← коллекция `stores`
   - `products` ← коллекция `products`
   - `customers` ← коллекция `customers`
   - `purchases` ← коллекция `purchases`

### Как работает

1. Для каждой коллекции MongoDB создаёт свой топик Kafka (если не существует)
2. Читает документы порциями (батчами) для эффективной обработки
3. Преобразует документы в формат JSON
4. Применяет HMAC-хеширование к чувствительным полям (если указан ключ)
5. Отправляет сообщения в Kafka с правильными ключами партиционирования

### Когда использовать

- После загрузки данных в MongoDB (`load_to_mongo.py`)
- Для передачи накопленных данных в поток обработки Kafka
- При необходимости обновить данные в ClickHouse через Kafka

## 5. 📥 run_consumer.py

**Назначение:** Приём данных из Kafka и загрузка в ClickHouse (Consumer).

### Что делает скрипт

Скрипт подписывается на топики Kafka и записывает сообщения в таблицы raw-слоя ClickHouse:

1. **Подключение к Kafka** — соединяется с брокером и подписывается на топики
2. **Подключение к ClickHouse** — устанавливает соединение с базой данных
3. **Чтение сообщений из Kafka** — потребляет сообщения из всех указанных топиков
4. **Вставка данных в ClickHouse** — записывает каждое сообщение в соответствующую таблицу:
   - Топик `stores` → таблица `raw.stores`
   - Топик `products` → таблица `raw.products`
   - Топик `customers` → таблица `raw.customers`
   - Топик `purchases` → таблица `raw.purchases`

### Как работает

1. Использует Consumer Group для координации между несколькими консюмерами
2. Автоматически фиксирует прочитанные сообщения (offset commit)
3. Поддерживает два режима работы:
   - **Непрерывный режим** — работает постоянно, обрабатывая новые сообщения
   - **Разовый режим** (`--once`) — обрабатывает накопленные сообщения и завершает работу

### Режимы работы

- **Постоянная подписка** (по умолчанию) — бесконечный цикл ожидания новых сообщений
- **Однократная обработка** (`--once`) — обработать доступные сообщения и выйти
- **С таймаутом** — ограничение времени работы в разовом режиме

### Когда использовать

- После запуска Producer (`run_producer.py`)
- Для непрерывной загрузки данных из Kafka в хранилище
- При необходимости обработать накопленные сообщения в Kafka

## 6. ⚡ run_etl.py

**Назначение:** Построение витрины признаков клиентов для ML-кластеризации.

### Что делает скрипт

Скрипт запускает ETL-процесс на базе Apache Spark для расчёта 30 бинарных признаков клиентов:

1. **Чтение данных из ClickHouse** — загружает факты покупок и справочники из mart-слоя
2. **Обработка на Spark** — выполняет сложные вычисления для расчёта признаков:
   - Анализ покупательских предпочтений (категории товаров, органические продукты)
   - Расчёт поведенческих метрик (частота покупок, средний чек)
   - Определение паттернов поведения (время покупок, предпочтения по магазинам)
   - Вычисление признаков лояльности и активности
3. **Сохранение результатов:**
   - В таблицу `mart.customer_features_mart` в ClickHouse
   - В файлы формата CSV (опционально)
   - В файлы формата JSON (опционально)
4. **Загрузка в S3** (опционально) — отправляет результаты в облачное хранилище S3

### Рассчитываемые признаки (30 штук)

**Предпочтения по продуктам:**
- Покупал молочные продукты за последние 30 дней
- Покупал фрукты/ягоды за последние 14 дней
- Не покупал овощи/зелень за последние 14 дней
- Предпочтение органическим продуктам
- Покупки хлеб/выпечка
- Покупки мясо/рыба/яйца за последние 7 дней
- Любитель фруктов (≥3 покупок за 30 дней)
- Вегетарианский профиль (нет мяса за 90 дней)

**Поведение при покупках:**
- Повторный покупатель (>2 покупок за 30 дней)
- Неактивный 14–30 дней
- Пользователь доставки
- Нет покупок (только регистрация)

**Лояльность:**
- Новый клиент (<30 дней с регистрации)
- Лояльный клиент (карта лояльности + ≥3 покупки)

**Траты:**
- Оптовый покупатель (средняя корзина >1000₽)
- Покупатель дешёвых товаров (<200₽)
- Крупные траты за последние 7 дней (>2000₽)

**Временные паттерны:**
- Ночной покупатель (после 20:00)
- Утренний покупатель (до 10:00)
- Выходного дня (≥60% покупок в выходные)
- Буднего дня (≥60% покупок в будни)
- Ранняя пташка (покупки 12:00–15:00)

**Предпочтения оплаты:**
- Предпочитает наличные (≥70% оплат)
- Предпочитает карту (≥70% оплат)

**Локация:**
- Покупки в разных городах
- Лояльность магазину (один магазин)
- Переключение между магазинами

**Корзина:**
- Покупатель одного товара (≥50% покупок — 1 товар)
- Разнообразный покупатель (≥4 категорий)
- Семейный покупатель (среднее кол-во позиций ≥4)

### Режимы вывода

- **ClickHouse** — запись только в базу данных
- **CSV** — выгрузка только в CSV-файлы
- **JSON** — выгрузка только в JSON-файлы
- **All** (по умолчанию) — все форматы одновременно

### Когда использовать

- После загрузки данных в mart-слой
- Для подготовки данных к ML-кластеризации клиентов
- При необходимости обновить витрину признаков
- Для выгрузки данных в аналитические инструменты

## 7. 🧹 cleanup_all.py

**Назначение:** Полная очистка всех данных проекта (удаление без возможности восстановления).

### Что делает скрипт

Скрипт принудительно удаляет данные из всех компонентов системы:

1. **Очистка файловой системы:**
   - Удаляет все подпапки в директории `data/`:
     - `customers/` — JSON-файлы клиентов
     - `products/` — JSON-файлы товаров
     - `purchases/` — JSON-файлы покупок
     - `stores/` — JSON-файлы магазинов
     - `spark/` — рабочие файлы Spark

2. **Очистка MongoDB:**
   - Удаляет базу данных `mongo_db` со всеми коллекциями

3. **Очистка Kafka:**
   - Удаляет топики: `stores`, `customers`, `products`, `purchases`

4. **Очистка ClickHouse:**
   - Удаляет базу данных `raw` со всеми таблицами
   - Удаляет базу данных `mart` со всеми таблицами и материализованными представлениями

### Как работает

1. Запрашивает подтверждение у пользователя (если не указан флаг `--yes`)
2. Последовательно выполняет очистку каждого компонента
3. Выводит отчёт об успешно удалённых данных
4. Продолжает работу даже если очистка одного компонента не удалась

### Режимы работы

- **Полная очистка** (по умолчанию) — удаляет все данные из всех источников
- **Выборочная очистка** — можно пропустить отдельные компоненты:
  - `--skip-mongo` — не очищать MongoDB
  - `--skip-kafka` — не очищать Kafka
  - `--skip-clickhouse` — не очищать ClickHouse
  - `--skip-data` — не очищать директорию `data/`
- **Автоматический режим** (`--yes`) — без запроса подтверждения

### Когда использовать

- Перед полным перезапуском пайплайна с чистыми данными
- Для освобождения дискового пространства
- При тестировании различных сценариев работы ETL
- **Внимание:** действие необратимо!

## 8. 🔄 dedup_mart.py

**Назначение:** Удаление дубликатов в таблицах mart-слоя ClickHouse.

### Что делает скрипт

Скрипт выполняет принудительную дедупликацию таблиц с помощью механизма `OPTIMIZE TABLE ... FINAL`:

1. **Анализ таблиц:**
   - Определяет движок хранения каждой таблицы
   - Подсчитывает количество строк и частей (parts)
   - Вычисляет количество дубликатов по ключевым полям

2. **Дедупликация:**
   - Применяет `OPTIMIZE TABLE ... FINAL` для объединения частей таблицы
   - Движок `ReplacingMergeTree` удаляет дубликаты строк при слиянии
   - Опционально использует `DEDUPLICATE BY` для более точного контроля

3. **Отчётность:**
   - Выводит статистику до и после обработки
   - Показывает количество удалённых дубликатов
   - Сохраняет отчёт в формате JSON (опционально)

### Обрабатываемые таблицы

**Измерения:**
- `dim_manufacturer` — производители
- `dim_store_location` — локации магазинов
- `dim_delivery_address` — адреса доставки
- `dim_product` — продукты
- `dim_customer` — клиенты
- `dim_store` — магазины

**Факты:**
- `fact_purchases` — факты покупок
- `fact_purchase_items` — позиции чеков

**Витрины:**
- `customer_features_mart` — признаки клиентов

### Режимы работы

- **Анализ** (`--analyze-only`) — показать статистику дубликатов без удаления
- **Дедупликация** (по умолчанию) — выполнить удаление дубликатов
- **Сухой запуск** (`--dry-run`) — показать план действий без выполнения
- **DEDUPLICATE BY** (`--deduplicate-by`) — использовать более точный метод дедупликации
- **Сохранение отчёта** (`--save-report`) — записать результаты в JSON-файл

### Стратегии дедупликации

1. **OPTIMIZE FINAL** (по умолчанию):
   - Объединяет все части таблицы в одну
   - `ReplacingMergeTree` автоматически удаляет дубликаты по версии
   - Быстрее, но менее гибко

2. **DEDUPLICATE BY** (опционально):
   - Явно указывает ключи для дедупликации
   - Учитывает колонки из `sorting_key` и `partition_key`
   - Точнее, но может работать дольше

### Когда использовать

- После множественных вставок данных в таблицы
- При обнаружении дубликатов в отчётах Grafana
- Для оптимизации производительности запросов
- Как плановая процедура обслуживания хранилища

## 🔧 Общие возможности всех скриптов

### Логирование

Все скрипты используют централизованную систему логирования:
- Логи сохраняются в папку `logs/`
- Каждый скрипт имеет свой лог-файл
- В логах отображается время, уровень сообщения и текст

### Конфигурация

Параметры подключения берутся из переменных окружения (файл `.env`):
- MongoDB: URI, имя базы данных
- ClickHouse: хост, порт, пользователь, пароль, имя базы
- Kafka: адрес брокера, consumer group
- S3: endpoint, bucket, ключи доступа

### Аргументы командной строки

Каждый скрипт поддерживает:
- Именованные параметры (`--parameter value`)
- Краткую форму для некоторых параметров (`-p value`)
- Флаги (`--flag` для включения опции)
- Справку (`--help` или `-h`)

## 📝 Примеры использования

### Полный запуск пайплайна

```bash
# 1. Инициализация таблиц
python scripts/init_clickhouse.py --confirm

# 2. Генерация данных
python scripts/generate_data.py

# 3. Загрузка в MongoDB
python scripts/load_to_mongo.py

# 4. Producer в Kafka
python scripts/run_producer.py

# 5. Consumer из Kafka
python scripts/run_consumer.py --once

# 6. ETL витрины
python scripts/run_etl.py --output-format all

# 7. Дедупликация (опционально)
python scripts/dedup_mart.py --force
```

### Очистка и перезапуск

```bash
# Полная очистка
python scripts/cleanup_all.py --yes

# Повторный запуск с начала
python scripts/generate_data.py
python scripts/load_to_mongo.py
# ... и так далее
```

### Анализ дубликатов

```bash
# Показать статистику дубликатов
python scripts/dedup_mart.py --analyze-only

# Удалить дубликаты с отчётом
python scripts/dedup_mart.py --force --save-report
```

## ⚠️ Важные замечания

1. **Порядок выполнения** — скрипты должны выполняться в определённом порядке из-за зависимостей между этапами

2. **Подтверждение действий** — скрипты `init_clickhouse.py`, `cleanup_all.py` и `dedup_mart.py` требуют подтверждения перед выполнением деструктивных операций

3. **Ресурсоёмкость** — ETL-процесс (`run_etl.py`) требует значительных ресурсов для работы Apache Spark

4. **Необратимость** — `cleanup_all.py` удаляет данные без возможности восстановления

5. **Логирование** — всегда проверяйте логи при возникновении ошибок
